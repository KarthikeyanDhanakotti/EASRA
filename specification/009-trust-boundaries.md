## 1. Purpose

This specification names, positions, and specifies the trust boundaries of EASRA. A trust boundary is a *named point at which identity, integrity, or confidentiality changes hands, and at which explicit controls apply*.

Trust boundaries are the enforcement points for Principle P2 (Security by Design). Threats from the OWASP LLM Top 10, MITRE ATLAS, and NIST AI RMF map to specific boundaries with named controls.

## 2. Boundary catalogue

EASRA defines four primary trust boundaries and two secondary planes.

| ID | Name | Position | Purpose |
|----|------|----------|---------|
| **TB-A** | Edge Boundary | L0 ↔ L1 | Untrusted external world → authenticated internal request. |
| **TB-B** | Response Boundary | L9 → L1 → L0 | Internal candidate → externally observable response. |
| **TB-C** | Model Boundary | L3/L6 ↔ model provider | Internal prompt → model provider (possibly external). |
| **TB-D** | Action Boundary | L7 → tool | Internal decision → real-world action / external system. |
| **TP-1** | Data Plane | L4, L5, L11 stores | Compute ↔ external stateful stores. |
| **TP-2** | Delivery Plane | L12 pipelines & registries | Change ↔ production surface. |

## 3. TB-A — Edge Boundary

**Position.** Between L0 (Channel) and L1 (Gateway/Identity).

**Trust transition.** Untrusted external caller → authenticated, rate-limited, session-bound internal request.

**Controls (all required).**

1. TLS termination at the edge; modern cipher suite policy.
2. WAF with rules for the API surface.
3. Bot protection with challenge/deny policy.
4. Authentication via the enterprise identity provider.
5. Coarse authorisation (does this identity have any right to this API?).
6. Rate limiting (per identity, per tenant, per channel).
7. Request schema validation.
8. Correlation-context injection (request ID, trace ID).

**Threats addressed.**
- OWASP LLM Top 10: *LLM04 Model DoS* (rate limits); *LLM10 Model Theft* (authn on model APIs).
- Classic: OWASP Web Top 10 injection, broken authentication, XSS via channel input.

**Failure modes.**
- IdP outage → cached tokens honoured within TTL; new sessions refused (P8).
- Rate-limit false positive → override channel via governance.

**Audit.** Every authn and authz decision is emitted to L13 audit log.

## 4. TB-B — Response Boundary

**Position.** Between L9 (Verification) → L1 → L0 (Channel).

**Trust transition.** Internal candidate response (possibly generated by a model, using retrieved evidence and tool outputs) → externally observable response.

**Controls (all required).**

1. Output guardrails (L8) applied before verification.
2. Verification verdict (L9) attached to every response.
3. Response formatter enforces client contract (schema, length, format).
4. PII policy applied on outbound content per data classification.
5. Response cache write-through respects identity scope.
6. Streaming responses have interim safety checks and end-of-stream verdicts.

**Threats addressed.**
- OWASP LLM Top 10: *LLM02 Insecure Output Handling* (formatter, schema); *LLM06 Sensitive Information Disclosure* (PII policy, redaction).
- Prompt-injection *exfiltration* (data-exfil via generated content) — prevented by output policy and format enforcement.

**Failure modes.**
- Verification failure → escalation path (S6).
- Formatter outage → safe fallback response.

**Audit.** Every emitted response has a verdict and correlation to grounding evidence.

## 5. TB-C — Model Boundary

**Position.** Between L3/L6 (internal) and the model provider (which may be external SaaS or internal inference).

**Trust transition.** Internal, potentially sensitive prompt → external inference surface.

**Controls (all required).**

1. Secrets never appear in prompts (enforced at L3).
2. PII classification (L3) drives routing policy (L6) — restricted classes must not go to unapproved providers.
3. Prompts marked with clear system / user / tool boundaries — mitigates prompt injection at model boundary.
4. Provider-specific authn scoped to L6 only; never exposed upstream.
5. Model registry (L14) enumerates approved models per data class and tenant.
6. Egress network policy allows only approved provider endpoints.

**Threats addressed.**
- OWASP LLM Top 10: *LLM01 Prompt Injection* (structural boundaries); *LLM03 Training Data Poisoning* (out of runtime scope, but model registry excludes unapproved models); *LLM05 Supply Chain* (approved providers only).
- Data exfiltration to unapproved third-party model.

**Failure modes.**
- Provider outage → fallback model per L6 policy.
- Rate limit → routed to alternate provider.

**Audit.** Every model call is logged with provider, model, tokens, cost, data-class routing decision.

## 6. TB-D — Action Boundary

**Position.** Between L7 (Tool Router) and the tool (MCP server, enterprise API, or external system).

**Trust transition.** Internal decision to act → real-world side effect.

**Controls (all required).**

1. Tool registry (L14) enumerates every tool, its schema, its impact class.
2. Tool-argument guardrails (L8) inspect arguments for policy violations.
3. Impact-class enforcement (P7): `write` requires authorisation; `high-impact` requires approval.
4. Caller identity is propagated to the tool (never impersonated by a service account for write/high-impact actions).
5. Every side effect is audited with actor, tool, arguments (redacted), result, evidence.
6. Idempotency where the tool supports it — retries must not duplicate side effects.
7. Time-boxed execution with timeout and cancellation.

**Threats addressed.**
- OWASP LLM Top 10: *LLM07 Insecure Plugin Design* (schema, argument guardrails, impact class); *LLM08 Excessive Agency* (impact class, approval).
- MITRE ATLAS: *ML.T0043 Adversarial Function Invocation*.
- Classic: SSRF via tool arguments, injection via tool output.

**Failure modes.**
- Tool unavailable → structured error; agent decides.
- Approval timeout → fail closed.

**Audit.** Every tool invocation with side effects is recorded. Audit records are queryable by identity, tool, tenant, and time window.

## 7. TP-1 — Data Plane

**Position.** Between compute layers (L2, L3, L6, L7, L8, L9) and external stores (L4 memory, L5 indexes, L11 caches).

**Controls.**

1. Encryption in transit (TLS) and at rest for every store.
2. Identity-scoped queries — no store returns records outside the caller's authorisation.
3. Field-level encryption or tokenisation for Restricted-class data.
4. PII classification enforced on write.
5. Cache keys never contain PII in cleartext.
6. Store access is audit-logged for Restricted-class data.

**Threats addressed.**
- OWASP LLM Top 10: *LLM06 Sensitive Information Disclosure* (memory/cache leakage); training data poisoning of retrieval indexes (mitigated by source registry & provenance).
- Multi-tenant leakage.

## 8. TP-2 — Delivery Plane

**Position.** Between CI/CD (L12) and production runtime.

**Controls.**

1. Signed artefacts (containers, models, prompts).
2. SBOM (software bill of materials) per release.
3. Prompt and model registry entries required before deployment.
4. Evaluation gates block promotion on regression.
5. Secretless pipelines (no long-lived credentials in the build).
6. Provenance metadata (SLSA-style) attached to every artefact.

**Threats addressed.**
- OWASP LLM Top 10: *LLM05 Supply Chain* (SBOM, signing); malicious prompt injection via pipeline (prompt registry + review).

## 9. Threat mapping (summary)

| Threat | Primary boundary | Named control |
|--------|------------------|---------------|
| LLM01 Prompt Injection | TB-A, TB-C, TB-D | Input guardrails, structural prompt boundaries, tool-arg guardrails |
| LLM02 Insecure Output Handling | TB-B | Output guardrails, formatter, schema enforcement |
| LLM03 Training Data Poisoning | (out of runtime) | Model registry (approved models only) |
| LLM04 Model DoS | TB-A, TB-C | Rate limits, per-tenant quotas, model timeouts |
| LLM05 Supply Chain | TP-2, TB-C | Signed artefacts, SBOM, approved providers |
| LLM06 Sensitive Info Disclosure | TB-B, TP-1 | PII policy, redaction, cache-key policy |
| LLM07 Insecure Plugin Design | TB-D | Tool registry, argument guardrails, impact class |
| LLM08 Excessive Agency | TB-D | Impact class, approval, audit |
| LLM09 Overreliance | TB-B | Verification verdict on every response, human-review escalation |
| LLM10 Model Theft | TB-A, TB-C | Authn on API and model endpoints, egress policy |

MITRE ATLAS and NIST AI RMF mappings are added in the handbook chapter on Security (L13).

## 10. Compliance mapping (summary)

| Standard | Where EASRA addresses it |
|----------|--------------------------|
| GDPR (data subject rights, minimisation) | TP-1 (classification, retention), TB-B (redaction) |
| HIPAA (PHI handling) | TP-1 (encryption, access), TB-C (approved providers) |
| SOC 2 (security, availability, confidentiality) | All boundaries + audit |
| ISO/IEC 42001 (AIMS) | L14 registries + audit + policy engine |
| EU AI Act (risk classification, transparency) | L14 model registry (risk class), L9 verdict on every response |

## 11. Boundary observability

Every boundary emits:

- Traffic metrics (requests, bytes, latency).
- Decision metrics (allow, deny, require-approval).
- Failure metrics.
- Audit records for security-relevant decisions.

Dashboards for each boundary are part of the handbook chapter on Observability (L10).

## 12. Boundary evolution

Adding a new trust boundary is a **major** version change (see [GOVERNANCE.md](../GOVERNANCE.md)). Adding a new *control* to an existing boundary is a minor change if additive.
