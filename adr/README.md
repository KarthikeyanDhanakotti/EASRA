# Architecture Decision Records (ADRs)

Every significant architectural decision in EASRA is recorded as an ADR. ADRs are the mechanism by which the specification evolves: no change to a layer boundary, interface, principle, or trust boundary lands without an ADR.

The ADR process is defined in [`../GOVERNANCE.md`](../GOVERNANCE.md) and [`../CONTRIBUTING.md`](../CONTRIBUTING.md).

## Format

We use [MADR](https://adr.github.io/madr/) with EASRA-specific additions:

- **Status.** Proposed / Accepted / Superseded (by ADR-NNNN) / Rejected.
- **Deciders.** Named maintainers.
- **Consulted.** Domain reviewers and public commenters.
- **Public comment period.** Required 14 calendar days for structural changes.
- **Consequences.** Positive, negative, and neutral impacts.
- **Alternatives.** At least two alternatives evaluated.
